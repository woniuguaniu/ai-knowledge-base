# 软件工程产品研发 SOP：从拿到需求到完成产品

> 软件工程不是"把代码写出来"，而是把一个模糊需求变成**可验证、可交付、可维护、可迭代**的产品。  
> 最科学的流程不是靠灵感冲刺，而是建立一条完整证据链：每一步都有输入、输出、决策依据和验收证据。

---

## 一句话总览

**产品研发 = 需求判断 + 设计定稿 + 分步实现 + 自动验证 + 真实验收 + 上线反馈。**

如果只记一句话：

> 不要从"写代码"开始，要从"这件事值不值得做、做成什么样才算完成、用什么证据证明完成"开始。

---

## 全流程地图

可以把研发过程想成一条从"想法矿石"到"成品机器"的流水线：

```text
需求输入
  ↓
0. 价值判断：要不要做？是不是该自研？
  ↓
1. 需求澄清：谁用？解决什么问题？边界是什么？
  ↓
2. 产品 Spec：页面、流程、权限、状态、验收标准
  ↓
3. 技术设计：架构、数据模型、接口、安全、部署
  ↓
4. 实施计划：拆任务、排顺序、定义测试与提交点
  ↓
5. TDD 开发：先写失败测试，再写最小实现
  ↓
6. 质量门禁：单测 / 集成 / E2E / lint / typecheck / build
  ↓
7. 验收发布：按角色跑真实流程，上线、回滚、监控
  ↓
8. 反馈迭代：真实用户数据 → 下一轮需求判断
```

这条链路里任何一步缺失，都会变成后面更贵的返工。

| 缺哪一步 | 常见后果 |
|---|---|
| 不做价值判断 | 花几个月做了一个其实该用低代码的内部工具 |
| 不做需求澄清 | 开发中途不断改口径，所有人都觉得"对方没说清楚" |
| 不写验收标准 | 产品做完后无法判断是否完成，只能靠感觉吵架 |
| 不做技术设计 | 数据模型、权限、状态流转后期返工成本极高 |
| 不写实施计划 | 写到哪算哪，容易漏掉边界、测试、部署 |
| 不做测试驱动 | 改一个地方坏三个地方，靠人工点页面救火 |
| 不做质量门禁 | 本地能跑，线上崩；一个小改动带出大事故 |
| 不做上线反馈 | 上线后没人用、用错、慢慢腐烂，都没人知道 |

## 阶段 0：需求进入前，先判断要不要做

老板、客户、朋友说"做个系统"，第一反应不要是"用 Vue 还是 React"，而是问：

> 这到底是**值得自研的产品**，还是**应该先用现成工具验证的内部流程**？

### 0.1 三个核心问题

| 问题 | 目的 |
|---|---|
| 谁用？ | 内部员工、外部客户、管理者、运营人员，决定产品形态 |
| 为什么现在要做？ | 是真痛点、老板灵感、竞品压力，还是临时管理需求 |
| 不做会怎样？ | 判断优先级和投入上限 |

### 0.2 四种处理方式

| 处理方式 | 适合场景 | 输出 |
|---|---|---|
| 不做 | 价值不明确、用户很少、替代方案够用 | 记录原因，避免反复讨论 |
| 用 SaaS / 低代码 | 内部表单、审批、工单、CRM、巡检 | 2-4 周验证流程 |
| 做 PoC | 技术不确定、AI / 算法 / 性能风险高 | 验证核心可行性 |
| 做产品化自研 | 用户清晰、流程稳定、数据资产重要 | 进入正式研发流程 |

### 0.3 内部工具的特殊判断

内部工具最容易误判：看起来像"做个 App"，本质上可能只是"员工填表 + 流程流转 + 权限控制"。

```text
用户 ≤ 30 人
+ 流程还在变化
+ 核心是表单 / 审批 / 工单 / 看板
+ UI 不需要强品牌体验
  ↓
优先低代码，不要直接自研
```

这部分可以直接参考 [内部工具选型方法论](内部工具选型方法论.md)：它讲的是**要不要自研**，本文讲的是**确定要做之后怎么研发**。

## 阶段 1：需求澄清与业务建模

需求澄清不是"把客户的话抄一遍"，而是把自然语言翻译成工程能实现的模型。

### 1.1 需求澄清五件套

| 维度 | 要问的问题 | 产物 |
|---|---|---|
| 用户 | 谁在什么场景下用？ | 角色清单 |
| 目标 | 用户完成什么任务？ | 用户故事 / 流程图 |
| 数据 | 需要记录哪些对象？ | 实体清单 |
| 状态 | 每个对象怎么流转？ | 状态机 |
| 规则 | 哪些能做、不能做、必须做？ | 业务规则表 |

### 1.2 从"功能列表"变成"业务对象"

差的需求：

```text
我要一个维修管理功能。
```

好的业务建模：

```text
对象：维修工单
角色：前台 / 维修师傅 / 保洁 / 店长
状态：待分配 → 待接单 → 处理中 → 待核查 → 已完成 / 已驳回
关键字段：房间、问题描述、图片、负责人、截止时间、核查签字
权限：前台可登记，维修师傅只能处理分配给自己的工单，店长可看全部
验收：能完整跑通登记 → 分配 → 接单 → 完成 → 核查签字
```

### 1.3 业务建模的三个图

| 图 | 解决什么问题 | 例子 |
|---|---|---|
| 角色-权限矩阵 | 谁能看、谁能改、谁不能碰 | 管理员 / 前台 / 维修 / 财务 |
| 状态机 | 对象从生到死怎么流转 | 工单、订单、审批单 |
| 数据关系图 | 哪些表 / 对象互相关联 | 用户、房间、工单、附件 |

### 1.4 澄清阶段的退出标准

只有满足这些条件，才进入产品 Spec：

- 每个核心角色都有明确使用场景
- 每个核心对象都有状态流转
- 每条关键业务规则能写成"如果 A，则 B"
- 至少列出 5-10 条验收场景
- 明确哪些功能本阶段不做

## 阶段 2：产品 Spec 与验收标准

Spec 是研发的合同。它不是越厚越好，而是要把"做成什么样"写到可以验收。

### 2.1 Spec 应该包含什么

| 模块 | 内容 |
|---|---|
| 背景与目标 | 为什么做，解决什么问题 |
| 范围 | 本阶段做什么、不做什么 |
| 角色与权限 | 谁能访问哪些页面、字段、动作 |
| 页面与流程 | 用户如何完成关键任务 |
| 数据字段 | 每个对象有哪些字段、校验规则 |
| 状态流转 | 状态、触发动作、异常分支 |
| 通知规则 | 何时通知谁，通过什么通道 |
| 验收标准 | 具体到可测试的场景 |

### 2.2 好验收标准的格式

差的验收标准：

```text
维修模块能正常使用。
```

好的验收标准：

```text
前台创建维修工单，选择房间、问题类型、截止时间并上传图片后：
1. 工单状态为"待分配"
2. 店长能在工单列表看到该记录
3. 维修师傅在未分配前看不到该工单
4. 后端返回的工单详情不包含无权限字段
```

### 2.3 Spec 的风险清单

每份 Spec 都应该有风险表：

| 风险 | 为什么危险 | 应对 |
|---|---|---|
| 需求一次性太大 | 团队消化不了，上线失败 | 拆 MVP / 阶段 2 / 阶段 3 |
| 权限粒度太粗 | 敏感信息泄露 | 做字段级权限 |
| 平台能力限制 | 小程序、浏览器、第三方接口有限制 | 写入技术约束 |
| 附件和数据膨胀 | 服务器磁盘和带宽爆炸 | 上对象存储和归档策略 |
| 验收标准模糊 | 做完无法判断成败 | 改成可测试场景 |

## 阶段 3：技术选型与架构设计

技术设计的目标不是炫技，而是让系统能以合理成本长期运行。

### 3.1 技术选型的五个维度

| 维度 | 要问的问题 |
|---|---|
| 业务匹配 | 这个技术能不能自然表达业务模型？ |
| 团队能力 | 现在的人会不会维护？ |
| 生态成熟 | 文档、社区、组件、运维工具是否成熟？ |
| 成本 | 开发成本、服务器成本、学习成本、迁移成本 |
| 风险 | 安全、合规、平台限制、供应商锁定 |

### 3.2 架构设计要覆盖的内容

| 设计项 | 典型产物 |
|---|---|
| 系统架构 | 前端、后端、数据库、缓存、对象存储、推送通道 |
| 模块边界 | auth / users / orders / repair / dashboard |
| 数据模型 | ER 图、索引、唯一约束、软删除策略 |
| API 契约 | REST / GraphQL / OpenAPI、错误码、分页、鉴权 |
| 权限模型 | RBAC、ABAC、字段级权限 |
| 异常处理 | 错误返回格式、重试、幂等、冲突处理 |
| 部署架构 | Docker、Nginx、HTTPS、备份、监控 |

更细的架构原则可以接着读 [软件架构设计详解](软件架构设计详解.md)。

### 3.3 架构阶段最常见的错

| 错误 | 表现 | 修正 |
|---|---|---|
| 过度设计 | 5 人内部系统上微服务、K8s、分库分表 | 从单体 + 清晰模块边界开始 |
| 忽略权限 | 只做登录，不做数据范围和字段权限 | 需求阶段就建角色权限矩阵 |
| 忽略数据迁移 | 直接改表，线上数据无法升级 | 用迁移工具管理 schema |
| 忽略附件 | 图片视频存在本机 | 用 OSS / S3 类对象存储 |
| 忽略观测 | 线上出问题只能猜 | 日志、指标、健康检查先设计 |

## 阶段 4：实施计划与任务拆分

Plan 是把 Spec 变成可执行动作。好的计划像施工图，差的计划像口号。

### 4.1 任务拆分原则

| 原则 | 解释 |
|---|---|
| 小步提交 | 每个任务 0.5-2 天内能完成 |
| 垂直切片 | 尽量从 UI 到 API 到 DB 打通一个小闭环 |
| 先核心后边角 | 先状态机、权限、数据，再做导出、报表、皮肤 |
| 先风险后舒适区 | 先验证第三方接口、性能、权限等高风险项 |
| 每步有测试 | 没有测试的任务，不算工程任务 |

### 4.2 实施计划的标准格式

```text
任务 1：建立维修工单状态机
输入：Spec 中的维修状态流转表
改动：state_machine.py / test_state_machine.py
测试：非法状态转换必须失败；合法转换必须成功
完成证据：pytest 指定用例通过

任务 2：实现创建工单 API
输入：维修工单字段定义
改动：models.py / schemas.py / router.py / service.py
测试：权限、字段校验、创建成功、无权限失败
完成证据：集成测试通过 + OpenAPI 可见
```

### 4.3 任务顺序推荐

```text
1. 数据模型和迁移
2. 领域规则 / 状态机
3. 后端服务与 API
4. 权限与安全边界
5. 前端页面和交互
6. 通知、附件、异步任务
7. E2E 真实流程
8. 部署和运维脚本
```

## 阶段 5：测试驱动开发

TDD 的核心不是"测试多一点"，而是：

> 先证明测试能抓住问题，再写最少代码让它通过。

### 5.1 红绿重构

```text
RED：写一个失败测试
  ↓
确认失败原因正确
  ↓
GREEN：写最小实现
  ↓
确认测试通过
  ↓
REFACTOR：清理结构，保持测试通过
```

### 5.2 哪些东西最应该 TDD

| 类型 | 为什么 |
|---|---|
| 权限 | 错一次就是数据泄露 |
| 状态机 | 非法流转容易藏 bug |
| 价格 / 金额 / 库存 | 业务损失直接发生 |
| API schema | 前后端契约一变就坏 |
| 数据迁移 | 线上数据不可随便试 |
| 工具函数 | 小函数被很多地方复用 |

### 5.3 测试分层

```text
单元测试：测一个函数 / 状态机 / 权限判断
  ↓
集成测试：测 API + 数据库 + 权限
  ↓
E2E 测试：像真实用户一样点完整流程
  ↓
手工验收：业务角色按清单跑关键场景
```

| 层级 | 特点 | 不适合承担什么 |
|---|---|---|
| 单元测试 | 快、准、定位清楚 | 不证明系统整体能用 |
| 集成测试 | 能发现模块连接问题 | 不证明 UI 体验 |
| E2E | 最接近真实用户 | 不适合覆盖所有边界 |
| 手工验收 | 能发现流程和认知问题 | 不该替代自动化测试 |

## 阶段 6：质量门禁与自动化验证

质量门禁就是"不达标不能合并 / 不能发布"。

### 6.1 基础门禁

| 门禁 | 目的 |
|---|---|
| format | 代码风格统一，减少无意义 diff |
| lint | 抓明显错误和坏味道 |
| typecheck | 提前发现类型契约错误 |
| unit test | 保证核心函数行为 |
| integration test | 保证 API / DB / 权限连通 |
| e2e test | 保证关键用户流程 |
| build | 保证能生成可部署产物 |
| migration check | 保证数据库升级可执行 |

### 6.2 CI/CD 的位置

CI/CD 不是"高级运维玩具"，而是把质量门禁自动化。

```text
开发者 push
  ↓
CI 自动拉代码
  ↓
安装依赖
  ↓
lint / typecheck / test / build
  ↓
全部通过才允许合并
  ↓
CD 构建产物并部署到测试 / 生产环境
```

完整机制可以参考 [CI/CD 与 GitHub Actions](CI-CD与GitHub-Actions.md)。

### 6.3 完成声明必须有证据

不要说：

```text
应该好了。
```

要说：

```text
已运行：
- backend pytest：152 passed
- frontend typecheck：passed
- e2e repair workflow：passed
- docker build：exit 0

剩余风险：
- 真实 OSS 上传未验证，因为当前环境没有生产 AK/SK
```

这不是形式主义，而是防止"看起来完成"。

## 阶段 7：上线、回滚、监控

上线不是最后一步，而是产品第一次接触真实世界。

### 7.1 上线前检查表

| 检查项 | 证据 |
|---|---|
| 数据库已备份 | 备份文件路径 / 恢复演练记录 |
| 迁移可执行 | migration 命令 exit 0 |
| 环境变量完整 | `.env.example` 与生产配置逐项对照 |
| HTTPS 可用 | 浏览器 / curl 验证 |
| 健康检查可用 | `/health` 返回 OK |
| 日志可查 | 能按 request_id 找到请求 |
| 回滚方案存在 | 上一版本镜像 / tag / 数据库回滚策略 |
| 关键流程验收 | 角色清单逐项通过 |

### 7.2 发布策略

| 策略 | 适合场景 | 风险 |
|---|---|---|
| 直接发布 | 内部小系统、低并发、可快速回滚 | 出问题影响所有人 |
| 灰度发布 | 有用户分组、有监控能力 | 需要路由 / 开关支持 |
| 蓝绿发布 | 不能长时间停机的服务 | 资源成本更高 |
| 金丝雀发布 | 大规模系统逐步放量 | 运维复杂度高 |

### 7.3 监控三层

```text
系统层：CPU / 内存 / 磁盘 / 网络
应用层：接口耗时 / 错误率 / 队列堆积 / 任务失败
业务层：注册数 / 下单数 / 工单完成率 / 超时率
```

技术指标告诉你"机器有没有坏"，业务指标告诉你"产品有没有真的工作"。

## 阶段 8：真实用户反馈与下一轮迭代

MVP 不是"缩水版终点"，而是学习工具。

### 8.1 上线后要收集什么

| 数据 | 作用 |
|---|---|
| 哪些功能没人用 | 删除或延后，不要继续投入 |
| 哪些流程卡住 | 优先优化真实摩擦点 |
| 哪些字段总填错 | 字段设计或文案有问题 |
| 哪些权限经常申请 | 角色模型可能错了 |
| 哪些 bug 高频出现 | 进入下一轮质量改进 |

### 8.2 反馈进入下一轮研发

```text
线上反馈
  ↓
问题分级：P0 / P1 / P2 / P3
  ↓
需求复盘：是真需求还是个别抱怨？
  ↓
更新 Spec
  ↓
更新测试
  ↓
进入下一轮迭代
```

### 8.3 迭代不是无限加功能

每轮迭代都要问：

- 这个功能解决了哪个真实问题？
- 有没有更便宜的运营 / 流程 / 配置方案？
- 做完后用什么指标证明有效？
- 会不会让现有用户更难用？
- 是否值得增加长期维护成本？

## AI 协作下的工程 SOP

AI 能极大提高研发速度，但也会放大"表演性完成"风险：它能写出很像完成报告的文字，却未必真的跑过验证。

### AI 参与研发时的正确分工

| 阶段 | AI 适合做 | 人必须把关 |
|---|---|---|
| 需求澄清 | 整理问题、提炼角色、发现矛盾 | 判断业务真实优先级 |
| Spec | 草拟结构、补边界、写验收标准 | 确认范围与取舍 |
| 架构 | 给方案、列风险、做对比 | 拍板技术栈和成本 |
| 编码 | 执行小任务、补测试、改 bug | 审查关键设计和安全 |
| 验证 | 跑命令、读日志、整理证据 | 判断证据是否足够 |
| 发布 | 生成 checklist、检查配置 | 控制生产权限和回滚 |

### 防止 AI "看起来完成"的 5 条规则

| 规则 | 做法 |
|---|---|
| 每个任务先写完成证据 | 没有证据的任务不准开始 |
| 让 AI 贴命令和结果 | 不接受"应该可以" |
| 大任务拆小 | 每个小任务能独立验证 |
| 关键路径人工复核 | 权限、金额、数据迁移、发布 |
| 保留真实验收清单 | 最终以业务流程跑通为准 |

这部分和 [LLM 典型失败模式](../06_Agent工程/LLM典型失败模式.md) 是同一条思想：**讲得像完成，不等于真的完成**。

## 用当前民宿运营系统做一次映射

以"民宿运营管理系统"为例，科学研发流程应该这样映射：

| 阶段 | 在该项目里的产物 |
|---|---|
| 价值判断 | 先判断这是内部 OA / 工单系统，不是 C 端 App |
| 范围切分 | MVP 做登录、房间、维修、整改、首页、推送；发票 / 水电气 / 房租放阶段 2 |
| Spec | 需求规格、技术架构、数据模型、API、UI、权限、测试文档 |
| 架构 | FastAPI + PostgreSQL + Redis + uni-app + Docker Compose |
| 计划 | W1-W8 分阶段完成报告 |
| TDD | 后端 unit / integration tests，前端工具函数测试 |
| E2E | Playwright 覆盖登录、房间、维修、整改、通知 |
| 验收 | MVP 验收清单按角色跑真实流程 |
| 发布 | DEPLOY 文档、Docker、Nginx、备份脚本 |
| 反馈 | 阶段 2/3 根据真实使用情况继续排期 |

关键不是这套技术栈本身，而是它体现了一个原则：

> 每个工程决策都能追溯到业务风险，每个完成声明都能落到测试或验收证据。

## 每阶段输入 / 输出 / 验收证据速查

| 阶段 | 输入 | 输出 | 验收证据 |
|---|---|---|---|
| 0 价值判断 | 原始想法 / 业务痛点 | 做 / 不做 / 低代码 / PoC / 自研决策 | 决策表 + 成本估算 |
| 1 需求澄清 | 访谈 / PDF / 口述需求 | 角色、流程、对象、规则 | 业务方确认无重大遗漏 |
| 2 产品 Spec | 澄清结果 | 页面、流程、字段、状态、验收标准 | 每条需求有可测场景 |
| 3 技术设计 | Spec | 架构、数据模型、API、权限、部署方案 | 风险表 + 技术取舍说明 |
| 4 实施计划 | Spec + 架构 | 任务清单、文件清单、测试计划 | 每个任务有完成证据 |
| 5 TDD 开发 | 计划任务 | 测试 + 实现 + 小提交 | 红绿重构记录 / 测试通过 |
| 6 质量门禁 | 代码变更 | CI、测试、构建、检查结果 | 全量门禁通过 |
| 7 上线发布 | 可部署产物 | 生产版本、部署记录、回滚点 | 健康检查 + 验收清单 |
| 8 反馈迭代 | 真实使用数据 | 下一轮需求池 | 指标 / 用户反馈 / 缺陷分级 |

## 30 秒口令版

```text
先定性：该不该做、要不要自研
再澄清：谁用、做什么、边界在哪
写 Spec：页面、流程、数据、权限、验收
做设计：架构、模型、接口、安全、部署
拆计划：小任务、测试先行、逐步提交
写代码：红绿重构，不靠感觉
跑门禁：lint / typecheck / test / build / e2e
再验收：按真实角色跑流程
才上线：备份、迁移、健康检查、回滚
看反馈：数据说话，进入下一轮
```
